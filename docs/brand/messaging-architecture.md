# Messaging Architecture

Last updated: 2026-06-28

---

## CT-1: Mission / Vision / Projects だけでは何が足りないか

Mission と Vision は「どこへ向かうか」を伝える。
Projects は「何を作ったか」を伝える。

しかし訪問者が最初に知りたいのは「なぜこの会社が存在するか」であり、
以下の要素がなければ理解は半分で止まる。

| 欠けている要素 | 説明 |
|--------------|------|
| **Origin (なぜ)** | 創業者がなぜ「縁」という名前を選んだか。個人的な原体験 |
| **Stakes (何が懸かっているか)** | ENISHIがなければ何が失われるか。存在の必然性 |
| **Connective Thread (束ねる軸)** | Family Quest と ご縁ノート が同じ会社のプロダクトに見える理由 |
| **Stance (AIに対する立場)** | AIを否定するのか、肯定するのか。ENISHIはどこに立つのか |
| **Emotional Payoff (感情的な報酬)** | ユーザーはどんな気持ちになるか。機能ではなく体験 |
| **Credibility (なぜ信頼できるか)** | この小さなスタートアップがなぜこれを作れるのか |

**結論:** Mission + Vision + Projects は「論理」を伝えるが、「感情」と「文脈」が欠ける。
Messaging Architecture はこのギャップを埋める。

## CT-3: 最上位メッセージ評価

**評価対象の思想:**
「AI時代に人間らしく生きるためのプロダクトを作る会社」

これはMission・Vision・Projectsを貫く軸になるか。

| テスト | 評価 |
|--------|------|
| Family Quest に当てはまるか | ✅ 家族という縁をゲームで深める = 人間らしさの体現 |
| ご縁ノート に当てはまるか | ✅ 大切な縁を記録・育てる = AIが替えられない人間的行為 |
| 将来の未知プロダクト に当てはまるか | ✅ "縁" という基準で選別できる |
| 競合と差別化できるか | ✅ "AIを作る"ではなく"AIで縁をつなぐ"は明確な差異 |
| 5秒で伝わるか | △ そのままでは長い → Core Messageで圧縮する |

**判定:** 思想の軸として有効。ただしWeb上で直接使うには長い。
Core Messageで「縁を、深く。」に圧縮し、この思想をその背景に置く。

## Brand Core

### Internal Belief (変えない核心)

> テクノロジーは、人と人の縁をつなぐためにある。

これはCompany OS・Mission・Visionから引き出される単一の文。
Web上には直接出さない。すべてのメッセージの根拠として機能する。

### Core Message (外向け・Heroに置く)

> **縁を、深く。**

- 「縁」= 会社名であり主題
- 「深く」= 表面的なつながり（SNS）への対比。ENISHIは深さを追求する
- 二語。圧縮された哲学。タイポグラフィで力を発揮する形

English equivalent: **"Connections, deeper."**
または: **"Technology for human connection."**

### Brand Tagline (サブヘッド・補足に使う)

> AI時代に、人間らしさのために。

- 文脈（AI時代）を設定し、ENISHIの立場（人間らしさ）を明示
- Core Messageとペアで使う

---

### Hero Copy Structure（CT-1への回答）

**問い:** 「縁を、深く。」だけで、初見ユーザーは「何の会社か」を理解できるか？

**答え: できない。** 詩的な力はあるが、それだけでは以下が伝わらない:
- 会社の種類（プロダクト会社？サービス会社？）
- 具体的に何を作っているか
- 誰のためか

**3層構造で補完する:**

```
H1:   縁を、深く。
      ↑ 感情的フック。会社名 = メッセージ

H2:   人と人をつなぐプロダクトを、AI時代に届ける。
      ↑ What (プロダクト会社) + When (AI時代) を1文で

Body: テクノロジーを、縁のために使う会社です。
      ↑ ENISHIの立場を1文で明確化。AIへの誤解を防ぐ
```

**設計原則:**
- H1は変えない（Core Messageは維持）
- H2で「会社の種類」と「時代背景」を与える
- Bodyは1文のみ。説明過多にしない
- この3層で「5秒の理解」を達成する

## Messaging Architecture（4層）

### Layer 1: Core Message

```
縁を、深く。
```

使用場所: Hero H1 / OGP / SNSプロフィール / 名刺

---

### Layer 2: Supporting Messages

訪問者が「それってどういうこと？」と感じた瞬間に答える3本の柱。

**S1 — 思想の説明**
> テクノロジーは、効率のためだけにあるのではない。
> 人と人をつなぎ、その縁を深めるためにある。

使用場所: Heroサブヘッド / About Overview

**S2 — 立場の表明（CT-2修正済み）**
> テクノロジーを、縁のために。
> AIを含むあらゆる技術を、人と人の縁を深めるために使います。
> テクノロジーは手段。縁が、目的です。

使用場所: About Mission / Topページ Vision セクション

**[修正理由]** 旧表現「私たちはAIを作る会社ではありません」は:
- 否定的定義（ポジティブに言い直す）
- MGX・AI Employee・Company OSとの矛盾を生む
- 「AIを使わない」という誤解を招く

新表現は「AIを使う、しかし目的は縁」という正確な立場を表現する。

**S3 — プロダクトの束ね方**
> Family Questで家族の絆を育て、ご縁ノートで出会いを記録する。
> すべてのプロダクトは、一つの信念から生まれています。

使用場所: /projects intro / About Snippet

---

### Layer 3: Proof Points

Supporting Messagesを裏付ける事実。

| Proof Point | 根拠 | 使用場所 |
|------------|------|---------|
| Family Quest — 家族の縁を育てるプロダクト | プロダクト自体 | /projects/family-quest |
| ご縁ノート — 出会いの縁を記録・育てるプロダクト | プロダクト自体 | /projects/goen-note |
| **設計思想の一貫性** | Company OSを外部翻訳。「体系的な思想から生まれた組織」として表現 | /about |
| **開発姿勢の透明性** | 現状を隠さないStatusバッジ。ADR的な意思決定の痕跡が製品品質に出る | /projects |
| **品質への考え方** | 思想とプロダクトが一致している。コピーと実際のプロダクトが矛盾しない | サイト全体 |

**[CT-4注記] Company OS の外部翻訳方針:**
Company OSは内部OSのため、外部Webサイトでは直接説明しない。
その存在が生む信頼シグナルを以下の言語で翻訳する:

| Company OS の実態 | 外部向け翻訳 |
|-----------------|------------|
| Company OS = 体系的組織運営 | 「設計思想の一貫性」「思想から生まれたプロダクト」 |
| AI Employee = AI活用の内製化 | 「テクノロジーを縁のために使う姿勢」（直接言及しない） |
| Repository Steward = 思想とコードの誠実な管理 | 「開発姿勢の透明性」「品質への考え方」 |
| ADR・設計ドキュメント = 判断の言語化 | 言及しないが、製品の一貫性として滲み出る |

## Project Category Design（CT-3: 10〜20プロダクト拡張への備え）

### 設計方針

プロダクトをフラットに並べるだけでは、20プロダクト時代に破綻する。
ENISHIブランドの核「縁」を軸にしたカテゴリで束ねる。

### カテゴリフレームワーク: 「〇〇の縁」

「縁」をカテゴリ名に含めることで、どのカテゴリも自動的にENISHIブランドと接続する。
新カテゴリを追加しても「X の縁」という形式を守ることで一貫性を維持。

```
カテゴリ        プロダクト例            縁の対象
──────────────────────────────────────────────
家族の縁        Family Quest           家族・血縁
出会いの縁      ご縁ノート             友人・出会い・ご縁全般
仕事の縁        MGX (tentative)        同僚・仕事上のつながり
学びの縁        将来                   師弟・学習コミュニティ
地域の縁        将来                   近隣・地域コミュニティ
```

### プロダクトページでの実装

```
/projects で表示する際:
  - タグとして「家族の縁」「出会いの縁」を表示
  - フィルタリングに使用（Growth時）
  - カテゴリ別にセクション分け（10プロダクト以上になってから）

Content Collections の frontmatter:
  category: "家族の縁" | "出会いの縁" | "仕事の縁" | "学びの縁" | "地域の縁"
```

### 拡張性の確認

| テスト | 評価 |
|--------|------|
| 新プロダクト追加時にカテゴリが自然に決まるか | ✅ 「誰との縁か」を問えば決まる |
| ブランドと連動するか | ✅ 全て「〇〇の縁」なのでENISHI感が保たれる |
| ユーザーに伝わるか | ✅ 生活実感に根ざした言葉（家族/出会い/仕事） |
| 並列に並べて美しいか | ✅ 文字数のリズムが均一 |
